## 2. 프로그래밍 언어의 3요소

• **Syntax (문법)** : 문장의 형식이 맞는가 ?

예1)
```
> print(3 +)
SyntaxError: invalid syntax
```
Error 발생 원인 : **+** 연산자는 이항 연산자* 인데, 뒤에 더할 대상이 오지 않았기 때문이다.

(1) 이항 연산자(binary operation) : 두 개의 항으로부터 결과를 얻어내는 연산

• **Static Semantics (정적 의미):** 문법은 맞지만 말이 되는가?

예2)
```
>print('Helloworld'/5)
TypeError: unsupported operand type(s) for /: 'str' and 'int'
```

Error 발생 원인 : 문자열과 숫자 사이의 수학 연산은 불가능하기 때문이다.

• **Semantics (의미):** 의도한 대로 작동하는가?
 주의해야 할 점은 정상적으로 실행이 되더라도, 답이 틀릴 수 있다.
예를 들어 혼합 계산을 진행 시, **괄호**를 사용하지 않고 진행한다면 수학적 우선순위에 따라
덧셈이 곱셈에 후순위로 밀릴 수 있다.

예3)  
```
a = 10  
b = 20  
  
avg1 = a+b / 2  
avg2 = (a+b) / 2  
  
print('avg1 = ',avg1) #20.0 (False)  
print('avg2 = ',avg2) #15.0 (True)
```

출력이 서로 다른 이유 :  수학적 우선순위를 고려하지 못했기 때문이다.

|       |      |                |
| ----- | ---- | -------------- |
| 연산자   | 우선순위 | 설명             |
| ()    | 1순위  | 괄호 안을 가장 먼저 계산 |
| * , / | 2순위  | 곱셈과 나눗셈        |
| + , - | 3순위  | 덧셈과 뺄셈         |

**[Remark] TypeError와 SyntaxError의 차이**

- **SyntaxError**: 코드를 실행하기 전(Compile/Parsing time)에 발견되는 구조적 결함.
- **TypeError**: 실행 중(Runtime)에 객체의 타입이 연산에 부적합할 때 발생하는 논리적 결함.

# 3. 파이썬의 자료형(Objects & Types)

• **Scalar Objects** (더 쪼갤 수 없는 것):
	- **int** : 정수 (3)
	- **float** : 실수 (3.0)
	-  **bool** : 참/거짓 = 1/0  (**형 변환 시 True는 1, False는 0으로 취급된다**)
	- **NoneType** : 값 없음

• **Type Conversion** (형 변환): 
**형 변환**이란 데이터의 타입을 다른 타입으로 바꾸는 행위 전체를 통칭한다.
int를 float로 , float를 int로 , str을(를) int로 등등
python의 **형 변환**은 두가지가 존재한다.

- **암시적 형 변환 (Implicit)** : 파이썬이 **자동**으로 해주는 것

예1) 
```
> print(3+1.5)
4.5
```
Mechanism : 정수(int)가 표현 범위가 더 넓은 실수(float)로 승격(Promotion)됨

- **명시적 형 변환 (Explicit)** : **직접** 함수를 써서 바꾸는 것

예2)
```
>x = "10"  
>y = 5  

>print(int(x) + y)
15
```
Mechanism : 문자열로 할당한 변수 **x** 를 **int(정수형)** 으로 **명시적 형 변환**을 하여 계산하였다.
명시적 형 변환을 하지 않은 상태로 계산한다면 다음과 같은 결과가 도출 된다.

예2 - 1)
```
>x = "10"  
>y = 5  
  
>print(x+ y)
TypeError: can only concatenate str (not "int") to str
```
Error 발생 원인 : 변수 **x** 는 문자열로 할당되었고, 변수 **y** 는 int로 할당되었는데, 
앞에서 설명했듯 문자열과 숫자 사이의 수학 연산은 불가능하다.

### **명시적 형 변환** Example

```
>x1 = 3
>x2 = 3.0
>x3 = False

>x1a = float(x1) 
>x2a = int(x2)

>print('x1 type : ',type(x1))
>print('x2 type : ',type(x2))
>print('x3 type : ',type(x3))

>print('---------------Type Conversion-----------------')

>print('x1a : int to float')
>print('x1a type : ',type(x1a))

>print('x2a : float to int')
>print('x2a type : ',type(x2a))

---------------------------------------------------------------------------------------------------------------
x1 type :  <class 'int'>
x2 type :  <class 'float'>
x3 type :  <class 'bool'>

---------------Type Conversion-----------------

x1a : int to float
x1a type :  <class 'float'>
x2a : float to int
x2a type :  <class 'int'>
```

**type()** 함수는 해당 변수에 데이터 타입을 알려주는 함수다.


**[Remark 1] 객체의 가독성(Readability)과 엄밀성(Precision)**

- **`int` vs `float`**: 정수 `3`과 실수 `3.0`은 수학적으로 동일한 값이지만, 
  컴퓨터 내부에서 저장되는 **메모리 구조**와 **정밀도**가 완전히 다르다.

- 파이썬은 데이터의 손실을 방지하기 위해 정수와 실수의 혼합 계산 시 **더 넓은 범위인** 
  `float`로 결과값을 일치시킨다. (이것이 앞서 언급한 **암시적 형 변환**의 필연성이다.)

**[Remark 2] NoneType의 실전적 의미**

- `None`은 단순히 '비어있음'을 뜻하는 추상적 개념이 아니라, 메모리상에 존재하는 **하나의 구체적인 객체**이다.

- 함수가 명시적으로 결과를 반환하지 않을 때 파이썬은 자동으로 `None`을 할당한다. 
  즉, **"값이 없음이 확정된 상태"**를 나타내는 도구로 이해해야 한다.

**[Remark 3] bool의 수치적 본질**

- **`bool`** 타입은 `int`의 하위 클래스(Subclass)이다.

- 내부적으로 `True`는 `1`, `False`는 `0`이라는 **정수값**을 점유하고 있다. 
  따라서 `True + 1`과 같은 연산이 문법적으로 허용되지만, 
  이는 **Semantics(의미론적)** 측면에서 코드의 가독성을 해칠 수 있으므로 주의가 필요하다.

**[Check Point] 형 변환 시 데이터의 가역성**

- **`int` → `float`**: 데이터 손실이 없는 **안전한 변환** (예: `3` → `3.0`)

- **`float` → `int`**: 소수점 아래 숫자가 버려지는 **비가역적 데이터 손실** 발생 (예: `3.9` → `3`)

- **`str` → `int`**: 문자열 내부가 숫자로만 구성된 경우에만 허용되는 **조건부 변환**
  형식이 맞지 않을 경우 `ValueError`를 발생시켜 프로그램이 즉시 정지된다.

# 4. 변수(Variable)

- **수학적 변수**: 값을 담는 '그릇' 혹은 변화하는 '수' 그 자체.

- **CS적 변수**: 메모리상에 존재하는 특정 데이터(객체)를 가리키는 **상징적 이름(Identifier)**.

**핵심 메커니즘: 할당(Assignment)**
파이썬에서 `a = 10`이라는 문장은 "변수 `a`에 `10`을 집입한다"는 수동적 개념이 아니라, 
**"메모리에 생성된 정수 객체 `10`에 `a`라는 이름을 붙인다(Binding)"** 는 능동적 행위로 해석해야 한다.

**[Remark] 변수와 객체의 독립성**

- **변수(Variable)**는 데이터 그 자체가 아니라, 데이터를 찾아가기 위한 **이정표**에 불과하다.
- 동일한 객체에 여러 개의 변수 이름을 붙일 수 있으며(Aliasing), 변수가 가리키는 대상이 바뀌더라도 기존 객체는 메모리상에 독립적으로 존재한다.

**[Check Point] 할당 연산자 `=`의 비대칭성**

- 수학적 기호 `=`는 좌항과 우항의 동등(Equality)을 의미하지만, 프로그래밍에서의 `=`는 **우항의 결과를 좌항의 이름에 부여**하는 방향성을 가진다.
- 따라서 `x = x + 1`과 같은 식은 수학적으로 모순이나, CS 관점에서는 "기존 `x`가 가리키던 값에 `1`을 더한 새로운 객체를 생성하여 다시 `x`라는 이름을 부여한다"는 논리적 필연성을 갖는다.

**[Commentary] 변수 명명 규칙(Naming Convention)의 당위성**

- 변수명은 단순히 데이터를 구분하는 용도를 넘어, 
  해당 데이터가 프로그램 내에서 수행하는 **역할(Role)** 을 명시해야 한다.

- 무분별한 변수명 사용은 **Semantics(의미론적)** 오류를 추적하기 어렵게 만드는 주된 원인이 되므로, 
  논리적 일관성을 유지하는 이름을 선택하는 것이 실전적 코딩의 핵심이다.

# 4 - 1. **변수 재할당과 바인딩의 변화** (Changing Bindings)**

- **정의:** 이미 선언된 변수에 대입 연산자(`=`)를 재사용하여, 
  해당 변수가 참조하는 **객체(Object)**를 교체하는 행위.

- **메커니즘:** 새로운 객체가 생성되고 변수의 **참조 주소(Memory Address)**가 갱신되는 과정이다. 
  이 과정에서 기존에 참조하던 객체는 해당 변수와의 연결이 끊기며, 더 이상 접근 불가능한 상태가 된다.

---

**[Remark 1] 수학적 등치(Equality) vs 프로그래밍적 대입(Assignment)**

많은 학습자가 `x = x + 1`이라는 식에서 혼란을 겪는 이유는 수학적 사고와 프로그래밍적 사고를 분리하지 못하기 때문이다.

1. **수학적 해석** : `x = x + 1` 은 `0 = 1 ` 이라는 모순을 도출하는 방정식이며, 해가 존재하지 않는다.

2. **CS적 해석** : 우항(`x+1`) 의 연산을 먼저 수행하여 새로운 객체를 생성한 뒤, 
   그 결과를 좌항(`x`)의 식별자에 **바인딩(Binding)**하는 명령**이다. 
   즉, `=` 기호는 '같다'가 아니라 **'우항의 결과를 좌항에 부여한다' 는 철저한 방향성을 가진다

**[Remark 2] 바인딩의 독립성과 비연동성 (No Automatic Update)**

변수 간의 관계는 할당이 일어나는 **'그 시점'의 상태값만을 반영하며, 
사후적인 값의 변화가 이전에 계산된 결과에 소급 적용되지 않는다.

변수의 값이 바뀌어도, 이전에 그 변수를 사용해 계산해둔 결과값은 **변하지 않는다**

예1)
```
>radius = 2.2  
>print('1st radius',radius)  
  
>area = 3.14 * (radius ** 2)  # (1) 시점: area는 15.19...  
>radius = radius + 1    # (2) 시점: radius가 3.2로 재할당됨  
>print('2nd radius',radius)

1st radius 2.2
2nd radius 3.2
```
**[Analysis]**
- `radius`의 값이 (1)에서 (2)로 변했다고 해서, 
  이미 계산이 완료된 `area`의 값이 자동으로 갱신되는 것은 아니다.

- `area`라는 변수는 (1) 시점에서 계산된 **결과 객체**를 가리키고 있을 뿐, 
  `radius`라는 변수와 수식적으로 연결(Link)되어 있는 상태가 아니기 때문이다.

- **결론:** 상태를 동기화하기 위해서는 반드시 해당 연산 과정을 **재실행(Re-evaluation)하여 
  바인딩을 갱신해야 한다.
