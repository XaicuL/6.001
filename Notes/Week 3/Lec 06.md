
# **Recursion & Dictionaries**

## **# 1. RECURSION (재귀): 분할과 정복의 당위성**

재귀는 문제를 더 단순한 형태의 동일한 문제로 축소하여 해결하는 **'Divide-and-Conquer'** 전략의 정수다.

### **(1) [Analysis]: 재귀의 필수 설계 요소**

- **Base Case (기초 조건)**: 더 이상 문제를 쪼갤 수 없는 가장 단순한 상태다. 이는 무한 루프를 방지하고 연산을 종결시키는 필연적 **탈출구** 역할을 한다.
    
- **Recursive Step (재귀 단계)**: 문제를 더 작은 입력값으로 변형하여 자기 자신을 호출하는 과정이다. 이를 통해 복잡한 문제를 점진적으로 단순화한다.
    
```
>def fact(n):
>   if n == 1: # [1] Base Case
>      return 1
>          else: # [2] Recursive Step
>        return n * fact(n-1)
```

### **[Analysis]: 왜 재귀적으로 사고해야 하는가?**

- **[1] 탈출구의 확보**: $n=1$일 때 결과를 즉시 반환하여 무한 루프를 방지한다. 기초 조건이 없다면 프로그램은 메모리 한계까지 자기 자신을 호출하다 붕괴한다.
    
- **[2]문제의 축소**: $n!$을 구하기 위해 그보다 한 단계 단순한 문제인 $(n-1)!$의 해답을 요구한다. 이 과정에서 각 호출마다 새로운 **Scope**가 생성되어 독립된 $n$값을 유지한다.
    
- **[Remark]**: 재귀는 "어떻게 계산할 것인가"보다 "이 문제가 어떤 하위 문제로 구성되어 있는가"에 집중하게 만든다


### **(2) [Necessity] 재귀적 사고의 필연성**

- **기존 방식의 모순**: 반복문(Iteration)은 상태 변수(`result`, `i`)를 끊임없이 업데이트해야 하므로 논리가 비대해질 수 있다.
    
- **해결책**: 재귀는 문제를 자기 유사성(Self-similarity)의 관점에서 바라본다. 프로그래머 입장에서 논리가 훨씬 직관적이며, 복잡한 수학적 정의(팩토리얼, 피보나치 등)를 코드에 그대로 박제할 수 있게 해준다.

## **# 2. DICTIONARIES (사전): 커스텀 인덱스의 혁명**

데이터를 찾기 위해 정수 인덱스(`0, 1, 2...`)에만 의존해야 했던 리스트의 한계를 극복하기 위해 등장한 가변(Mutable) 객체다.

### **(1) 사전의 필연적 등장 배경**

- **기존 방식의 모순**: 이름, 성적, 과목 정보를 각각의 리스트에 담으면, 인덱스를 맞추기 위해 모든 리스트의 길이를 동기화해야 하는 관리의 지옥이 펼쳐진다. 데이터를 하나 찾을 때마다 매번 `index()`를 호출해야 하는 비효율성도 문제다.
    
- **해결책**: **Key(키)** 와 **Value(값)** 를 한 쌍으로 묶어, 정수가 아닌 '이름'이나 '라벨'로 데이터에 즉시 접근(Direct Lookup)한다. 단 한 줄의 코드로 원하는 정보를 정확히 추출할 수 있다.

```
>age = {'Jun' : '21' , 'nuj' : '24' , 'hyun' : '23' }  
  
>print(age['Jun']) #dictionary age 에서 Jun 이라는 정보를 찾아 결과를 출력  
  
>age['nuyh'] = '25' #age에 정보 추가  
>print(age) #정보가 추가 됨을 검증

21
{'Jun': '21', 'nuj': '24', 'hyun': '23', 'nuyh': '25'}

```
    
### **(2) [Analysis] : 사전의 엄밀한 규칙**

- **Keys**: 유일해야 하며(Unique), 반드시 **Immutable(불변)** 타입이어야 한다. 이는 내부적으로 해싱(Hashing)을 통해 주소를 고정하기 위함이다.
    
- **Values**: 중복이 허용되며, 리스트나 또 다른 사전 등 어떤 타입이든 담을 수 있는 높은 유연성을 가진다.
    
- **[Check Point]**: 사전은 요소 간의 **순서를 보장하지 않는다**. 데이터의 선후 관계가 중요하다면 리스트를 병행하여 사용해야 한다.

## **# 3. Memoization (메모이제이션)**

재귀의 우아함과 사전의 효율성을 결합한 최고의 최적화 기법이다.

- **모순**: 피보나치 수열을 단순 재귀로 풀면, 이미 계산한 값($fib(3)$ 등)을 수백만 번 중복 계산하는 치명적 낭비가 발생하며 연산 횟수가 지수적으로 폭발한다.
    
- **해결책 (Memoization)**: 사전을 사용하여 이미 계산된 결과값을 저장해두고, 동일한 호출이 발생할 때 즉시 꺼내 쓴다(Lookup).
    
- **[Effect]**: $fib(34)$ 계산 시, 호출 횟수가 **11,405,773번에서 단 65번**으로 압축되는 경이로운 성능 향상을 보여준다.
