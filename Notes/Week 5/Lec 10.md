---

## **# 1. 효율성 분석의 필연성**

컴퓨터의 속도가 빨라지고 있지만, 다뤄야 할 데이터셋의 크기는 그보다 훨씬 거대해지고 있다. 단순한 브루트 포스(Brute force) 방식은 데이터 규모가 커질 때 물리적인 시간 한계에 부딪히게 된다.

### **(1) [Necessity] 시간 vs 공간 복잡도**

- **Time Efficiency**: 프로그램이 연산을 수행하는 데 걸리는 시간.
    
- **Space Efficiency**: 프로그램이 실행되는 데 필요한 메모리 양.
    
- **트레이드오프(Trade-off)**: 메모리를 더 써서 시간을 단축(예: 메모이제이션)하거나, 시간을 더 들여 메모리를 아끼는 선택이 필연적으로 발생한다.
    

---

## **# 2. BIG "O" NOTATION: 성장의 질서**

알고리즘의 효율성을 측정할 때, 구체적인 실행 시간(초)은 하드웨어마다 다르므로 의미가 없다. 우리는 입력 크기($n$)가 무한히 커질 때 실행 시간이 늘어나는 **'성장률(Order of growth)'** 에 집중한다.

### **(1) Big O의 엄밀한 규칙**

- **상수항 무시**: $O(n + 100)$은 $O(n)$이다.
    
- **계수 무시**: $O(3n)$은 $O(n)$이다.
    
- **지배적인 항만 유지**: $O(n^2 + n)$에서 $n$은 $n^2$에 비해 미미하므로 $O(n^2)$으로 표기한다.
    
- **최악의 경우(Worst Case)**: Big O는 항상 알고리즘이 마주할 수 있는 가장 불운한 시나리오에서의 상한선(Upper bound)을 보장한다.

## **# 3. 주요 복잡도 클래스 (Complexity Classes)**

### **(1) 상수 시간: $O(1)$**

- 입력 크기에 상관없이 연산 횟수가 일정하다.
    ![[CleanShot 2026-03-02 at 20.32.51@2x.png]]

### **(2) 로그 시간: $O(\log n)$**

- 문제를 매번 절반으로 쪼개는 방식이다. (예: 이분 탐색)
    
- 데이터가 2배 늘어나도 연산은 1회만 늘어나는 경이로운 효율을 보여준다.
    ![[CleanShot 2026-03-02 at 20.36.26@2x.png]]

### **(3) 선형 시간: $O(n)$**

- 리스트의 요소를 하나씩 순회하는 방식이다.
    
- 입력 데이터 양에 비례하여 시간이 정직하게 증가한다.
    ![[CleanShot 2026-03-02 at 20.37.29@2x.png]]

### **(4) 다항 시간: $O(n^c)$**

- **이차 시간 ($O(n^2)$)**: 중첩 루프(Nested loops)를 돌릴 때 발생한다.
    
- 데이터가 조금만 커져도 실행 시간이 기하급수적으로 폭발하기 시작하는 '위험 구간'이다.

## **# 4. 중첩 루프의 분석**

가장 많이 실수하는 지점은 중첩된 루프의 범위를 단순 합산하는 것이다.


예)
```
def g(n):
    x = 0
    for i in range(n):     # n번 반복
        for j in range(n): # n번 반복
            x += 1
    return x
```

- **[Analysis]**: 안쪽 루프가 $n$번 돌고, 그것을 감싸는 바깥 루프가 다시 $n$번 돌기 때문에 
  총 연산 횟수는 $n \times n = n^2$이 된다. 따라서 이 알고리즘의 복잡도는 **$O(n^2)$**이다.
